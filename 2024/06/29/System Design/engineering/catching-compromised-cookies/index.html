Catching Compromised Cookies

슬랙의 쿠키가 도난당했을 때 이를 감지하고 경고하는 방법에 대해 살펴보자.

Detecting cookie misuse

핵심 아이디어는 session forking을 감지하는 것이다. 즉, 여러 기기에서 동시에 쿠키가 사용되고 있는지 이해하는 것이다.

슬랙이 세션 분기를 감지하기 위해 핵심적으로 사용되는 것은 마지막 액세스 타임스탬프다.

Last access timestamp

마지막 액세스 타임스템프를 서버에 저장해두고, 이후 요청시 들어오는 쿠키의 타임스탬프를 데이터베이스의 타임스탬프와 비교해 일치 여부를 판단한다.

슬랙은 쿠키의 마지막 액세스 타임스탬프를 접근이 발생할 때마다 업데이트 하게 되는데, 공격자가 도난당한 오래된 타임스탬프가 있는 쿠키를 획득하면 마지막 액세스 타임스탬프가 일치하지 않기 때문에 fork가 발생한 것을 알 수 있다. 이를 방지하려고 공격자가 쿠키를 슬랙과 정기적으로 상호작용한다고 했을 때, 원래 사용자가 슬랙을 다시 시작하면 슬랙 데이터베이스의 쿠키 사본과 타임스탬프가 일치하지 않기 때문에 어느 쪽이 합법적인지는 알 순 없지만 복사본이 발생했 다는 것은 감지할 수 있게 된다.

Testing

False positives

정상적인 사용자가 업데이트된 타임스탬프를 사용하는 쿠키를 응답받지 못하는 경우도 발생하는데, 이는 클라이언트의 타임스탬프와 데이터베이스의 타임스탬프가 달라 도난당한 쿠키와 같이 취급되는 잘못된 감지 이벤트가 발생하는 경우다.

그래서 슬랙은 IP 주소를 참조하게 함으로써 오탐율이 상당 부분 제거되었으나, 설계의 일부 주요 단점을 해결하지는 못했다. 마지막 액세스 타임스탬프가 정상적으로 작동하기 위해서는 클라이언트가 쿠키를 안정적으로 설정해야 한다는 문제가 있었는 이를 해결하기 위해 멱등적인 2 phase 접근방법을 채용한다.

  • 별도의 세션 후보 쿠키를 설정해 세션 쿠키 업데이트
  • 새 세션 후보 쿠키가 설정된 요청을 받은 클라이언트는 세션 쿠키로 승격함
  • 클라이언트가 세션 후보 쿠키를 통해 더 최신의 타임스탬프를 제공한 후 데이터베이스의 타임스탬프를 업데이트 함

이 접근 방식의 시나리오에서 클라이언트가 특정 요청에 대한 응답을 받지 못하면 프로세스가 중단된 부분부터 다시 시작한다.

또한 일부 클라이언트가 일련의 API 요청을 빠르고 연속적으로 보내는 경쟁 조건의 영향을 완화하기 위해 데이터베이스의 쿠키 타임스탬프가 방금 업데이트된 경우, 요청의 타임스탬프를 무시하는 전략을 사용한다. 공격자가 이런 허점을 뚫기 위해서는 데이터베이스가 업데이트되는 시기를 정확히 예측해야 하는데, 이는 매우 어려운 문제이기 때문이다.
