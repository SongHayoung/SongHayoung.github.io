Mastering the Art of API Design

3부로 구성된 이 시리즈에서는 API 설계에 대해 이야기한다.

  1. API First 원칙이 조직에 중요한 이유는 무엇인지?
  2. 효과적이고 안전한 API를 설계하려면 어떻게 해야 하는지?
  3. 생산성을 높일 수 있는 도구에는 어떤 것이 있나요?

API는 시스템 구성 요소가 서로 상호 작용하는 방식을 정의하는 일련의 프로토콜이다. 아키텍처 스타일이 진화함에 따라 최근 몇 년 동안 API가 인기를 얻고 있다. 아래 다이어그램은 마이크로 서비스와 클라우드 네이티브 애플리케이션의 부상으로 인해 서비스가 더욱 세분화되는 과정을 보여준다. 모놀리식 애플리케이션의 프로세스 내 호출은 마이크로 서비스 및 서버리스 애플리케이션의 프로세스 간 호출로 전환된다. 또한 각 프로세스가 서로 다른 물리적 서버에 상주할 수 있으며 다양한 네트워크 문제로 인해 서비스 호출이 실패할 수 있다.

서비스 복잡성이 증가함에 따라 보다 체계적인 API 설계의 필요성이 강조되고 있다.

API First

지난 10년 동안 API First는 인기있는 소프트웨어 개발 모델로 부상했다. 이 모델은 시스템 설계보다 API 설계에 우선순위를 둔다. 다양한 기능 팀과 시스템이 API를 공유 커뮤니케이션 언어로 사용한다. 이러한 API는 비즈니스 요구 사항과 시스템 설계를 위한 스펙 역할을 한다. 그런 다음 각 팀은 독립적으로 작업하고 개발 테스트 단계에서 다시 모인다.

아래 다이어그램은 Code First 접근 방식과 API First 접근 방식을 비교한 것이다. Code First 모델에서 API는 시스템 설계의 부산물이며 흔히 “문서화” 라고 한다. API First 모델은 API 스펙으로 시작해 API 중심 테스트로 마무리하므로 API가 전체 소프트웨어 개발 주기의 원동력이 된다.

API First는 몇가지 이점을 제공한다.

  1. 시스템 통합 개선 : API First는 개발자가 프로젝트 시작 단계부터 시스템 상호 작용을 신중하게 고려하도록 장려해 개발 중 지속적인 수정의 필요성을 줄여준다.
  2. 협업 및 품질 향상 : API는 조직 내에서 공유 사양 역할을 하므로 개발자, 테스터, DevOps가 독립적으로 작업할 수 있다. 프로젝트 초기에 API에 동의하면 불확실성을 제거하고 소프트웨어 품질을 높이는데 도움이 된다.
  3. 확장성 향상 : 각 서비스에 대해 정의된 인터페이스를 사용하면 새 인스턴스를 스핀업하고 로드 밸런서 설정을 조정해 확장을 더욱 쉽게 관리할 수 있다.

API First 설계는 효율성과 투명성 뿐만 아니라 네트워킹 효과도 촉진시킨다. 모든 시스템을 확장성과 유연성을 염두에 두고 설계하도록 의무화하는 것은 매우 선구적인 발상이다. 그 결과 회사는 변화하는 비즈니스 환경에 신속하게 적응할 수 있게 된다.

API Architectural Styles

API 아키텍처 스타일마다 서로 다른 통신 프로토콜과 데이터 형식을 사용한다. 일반적인 스타일에 대한 개요는 아래 다이어그램에 나와있다.

1. REST

2000년에 로이 필딩에 의해 소개된 REST(Representational State Transfer)는 프론트엔드 클라이언트와 백엔드 서비스 간에 가장 널리 사용되는 스타일이다. RESTful 아키텍처에서 모든 구성 요소는 리소스이며, GET, POST, PUT, DELETE와 같은 표준 HTTP 메서드를 사용해 엑세스한다. 페이로드 형식은 JSON, XML, HTML, 또는 plain text가 될 수 있다.

REST는 웹 서비스를 진정한 RESTful로 만드는 6가지 아키텍처 제약 조건을 정의한다.

  1. Uniform interface : 리소스에 대한 API 인터페이스를 정의해야 한다.
  2. Client-Server : 클라이언트 측 애플리케이션과 서버 측 애플리케이션은 별도로 발전해야 한다.
  3. Stateless : 모든 클라이언트-서버 상호 작용은 상태 비저장입니다. 서버는 모든 요청을 새로운 요청으로 처리한다.
  4. Cacheable : 데이터와 응답은 가능한 한 캐시된다.
  5. Layered system : API, 서비스, 데이터를 서로 다른 서버에 배포할 수 있다.
  6. Code on demand(optional) : 이 선택적 제약 조건을 사용하면 필요한 경우 서버가 실항 가능한 코드를 반환할 수 있다.

2. GraphQL

GraphQL은 2015년에 Meta에서 제안했다. 엔티티간에 그래프와 같은 관계가 있는 복잡한 시스템에 적합한 스키마와 타입 시스템을 제공한다. 중첩된 데이터 구조로 복잡한 쿼리를 처리한다. 예를 들어 아래 다이어그램에서 GraphQL은 한 번의 호출로 사용자 및 주문 정보를 검색할 수 있는 반면 REST는 서로 다른 엔드포인트를 여러 번 호출해야 한다.

GraphQL은 REST를 대체하는 것이 아니라 기존 REST 서비스를 기반으로 구축하므로 마이그레이션이 덜 침투적이다. 그러나 GraphQL은 새로운 복잡성을 가져온다. 예를 들어, 쿼리를 신중하게 설계하지 않으면 GraphQL은 필요 이상으로 많은 리소스 필드를 노출할 수 있다. 또한 쿼리 유연성이 증가하기 때문에 캐싱이 더 어려워진다. GraphQL을 도입하기 전에 필요성을 평가해야하며, 새로운 쿼리 언어와 새로운 스키마 설계를 이해하는 러닝 커브가 높다는 점도 인지해야 한다.

3. WebSocket

웹소켓은 TCP를 통해 전이중 통신을 제공하는 프로토콜이다. 클라이언트는 웹소켓을 설정해 백엔드 서비스에서 실시간 업데이트를 수신한다. 항상 데이터를 pull하는 REST와 달리 WebSocket은 데이터를 push할 수 있다. 온라인 게임, 주식 거래, 메시징 앱과 같은 애플리케이션은 실시간 통신을 위해 웹소켓을 활용한다.

4. Webhook

Webhook은 일반적으로 서드파티 비동기 API 호출에 사용된다. Webhook을 사용하면 한 애플리케이션이 다른 애플리케이션으로부터 업데이트를 받도록 등록할 수 있다. Saas가 대중화됨에 따라 SaaS 서비스 통합을 위해 Webhook사용이 널리 확산되었다. 많은 Webhook서비스가 API에 Webhook지원을 포함하고 있다.

예를 들어 아래 다이어그램에서는 결제 채널로 Stripe 또는 Paypal을 사용하고 결제 결과에 대한 Webhook을 등록한다. 서드파티 결제 서비스가 프로세스를 완료하면 결제 성공 또는 실패 여부를 당사 결제 서비스에 알린다. Webhook 호출은 일반적으로 시스템 상태 머신의 일부를 구성한다.

5. gRPC

Google에서 2016년에 출시한 gRPC는 분산 시스템에서 서버 간 통신에 사용되는 최신 오픈 소스 RPC(Remote Procedure Call) 프레임워크다. gRPC는 언어에 구애받지 않는 API를 제공하며 통신에서 데이터 직렬화를 위해 프로토콜 버퍼를 사용한다. REST와 비교했을 때, gRPC는 클라이언트 및 서버 스텁을 생성하는데 도움이 되는 코드 생성 도구를 제공해 데이터 전송을 위한 코딩 작업을 줄여준다. gRPC는 HTTP/2를 기반으로 하여 멀티플렉싱 및 스트리밍을 허용하므로 클라이언트와 서버가 동시에 데이터를 송수신할 수 있다.

RPC 프레임워크의 단점은 원격 프로시저 호출을 로컬 프로시저 호출처럼 보이게 하여 신뢰할 수 없는 네트워크 처리의 복잡성을 감춘다는 점이다. 개발자가 네트워크 문제로 인해 응답이 간헐적으로 끊길 수 있다는 사실을 인지하지 못하면 심각한 버그로 이어질 수 있다.

6. SOAP

SOAP(Simple Object Access Protocol)은 내부 시스템 간의 통신에 XML 페이로드를 사용한다.

7. Kafka

요청 - 응답 통신 메커니즘을 기반으로 하는 6가지 아키텍처 스타일을 살펴보았다. Kafka는 종종 구독 - 게시 기반 통신을 용이하게 하는 메시징 계층으로 사용된다. 이 이벤트 스트리밍 패러다임은 다음과 같은 측면에서 요청 - 응답 패러다임과 다르다.

위 다이어그램에서 주문 서비스는 일반적으로 결제 서비스가 주문 서비스보다 느리기 때문에 Kafka를 통해 결제 서비스로 주문을 전송한다. 결제에 REST API를 사용하면 응답을 기다리는 시간이 길어져 처리량에 영향을 미칠 수 있다. 또한 Kafka는 하나의 입력을 여러 출력으로 전송하는 fan-out을 지원한다. 요청 - 응답 패러다임이 fan-out에 사용된다면 코드베이스는 금방 유지 관리가 불가능해질 것이다.

Popularity

Postman은 각 아키텍처 스타일의 인기도에 대한 설문 결과를 발표했다. REST는 설문조사 참가자의 89%가 선택한 여전히 지배적인 스타일이다. Webhook(35%), GraphQL(28%), gRPC(11%)는 전년도에 비해 인기가 높아졌다. 이는 내부 마이크로서비스에 gRPC를 사용하고 서로 다른 데이터 원본을 연결하기 위해 GraphQL을 사용한것이 원인이다.

Comparison

지금까지 7가지 유형의 아키텍처 스타일에 대해 설명했다. 아래 다이어그램을 사용하여 몇 가지 인기 있는 스타일을 비교해 보자. 각 스타일은 특정 목적을 위해 개발되었다. 예를 들어 GraphQL은 복잡한 REST 기반 API간의 관계를 간소화하기 때문에 대기업에서 인기가 높아졌고, RPC는 기업들이 마이크로서비스 아키텍처를 더 널리 채택함에 따라 마이크로서비스의 표준 프로토콜이 되었다.

시스템을 설계할 때는 다양한 시나리오에 적합한 상호 작용 방법을 선택해야 한다. 일부 극단적인 상황에서는 이러한 아키텍처 스타일 중 어느것도 적합하지 않으며 독점적인 통신 프로토콜을 개발해야 한다. 이러한 요구사항은 RPC가 너무 느린 저지연 트랜잭션 애플리케이션에서 일반적이다.
