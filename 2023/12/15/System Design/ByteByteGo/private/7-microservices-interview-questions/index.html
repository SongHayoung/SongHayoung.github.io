7 Microservices Interview Questions

마이크로서비스는 오늘날 소프트웨어 시스템을 구성하는데 널리 사용되는 방법이다. 기업의 규모가 커지고 클라우드 컴퓨팅을 더 많이 사용함에 따라 마이크로서비스는 복잡성을 해결하는데 도움이 된다. 이번 포스트에서는 마이크로서비스의 주요 개념과 인터뷰에 자주 등장하는 질문을 살펴본다.

  1. What are microservices?
  2. What issues do microservices aim to solve?
  3. What new challenges do microservices introduce?
  4. What are some popular microservices solutions?
  5. How does monitoring and alerting work with microservices?
  6. How are logs collected and analyzed?
  7. What is a Service Registry?

1. What Are Microservices?

마이크로서비스의 주요 측면에 대해 Martin Fowler와 Adrian Cockcroft의 말을 인용한다.

From Martin Fowler:

“The microservice architectural style is an approach to developing a single application as a suite of small services, each running in its own process and communicating with lightweight mechanisms, often an HTTP resource API. These services are built around business capabilities and independently deployable by fully automated deployment machinery. There is a bare minimum of centralized management of these services, which may be written in different programming languages and use different data storage technologies.”

From Adrian Cockcroft:

“Microservices are loosely coupled service-oriented architecture with bounded contexts.”

Key Aspects of Microservices

Martin Fowler와 Adrian Cockcroft의 정의를 통ㅎ ㅐ마이크로서비스 아키텍처의 주요 측면을 아래와 같이 요약할 수 있다.

  1. 모놀리식 애플리케이션을 소규모의 독립적인 서비스로 분해한다. 이를 통해 다양한 제품 팀이 특정 비즈니스 역량에 맞는 서비스를 개발, 테스트 및 배포할 수 있다. 대규모 조직에서 모놀리식 시스템을 해체하고 생산성을 향상하는 데 유용하다.
  2. API를 통해 통신하는 느슨하게 결합된 서비스. 프론트엔드 / 백엔드 구성 요소는 REST를 통해 통신하고, 서비스 간 통신은 효율적인 요청 / 응답을 위해 RPC를 사용한다.
  3. 경계가 있는 컨텍스트를 중심으로 세심하게 설계된다. 각 서비스에는 명확한 모듈 경계와 캡슐화된 도메인 로직이 있어 서비스 간의 긴밀한 결합을 방지한다.
  4. 효과적인 DevOps 관행을 지원한다. 마이크로서비스 개발 방법론 중 중요한 부분이다. 소규모의 풀스택 팀이 특정 서비스를 end-to-end로 완전히 소유한다. 컨테이너화, 자동화, 컨테이너 오케스트레이션을 통해 마이크로서비스를 효과적으로 배포할 수 있다.
  5. 수평적 확장이 가능하도록 설계되어 장애에 대한 복원력이 뛰어나다. 필요에 따라 서비스를 독립적으로 확장할 수 있다. 내결합성을 염두에 두고 구축되었다.
  6. 분산형 거버넌스 및 유연성. 팀은 서비스에 적합한 기술을 선택할 수 있다.
  7. 아래와 같은 이유로 광범위한 모니터링 및 계측이 필요하다.
    1. 서비스 증가 - 모놀리스가 여러 개의 독립적인 서비스로 분해되면서 추적해야 할 구성 요소의 수가 빠르게 증가한다.
    2. 컨테이너 / 오케스트레이터를 통한 인프라 추상화 - k8s와 같은 런타임 플랫폼은 인프라를 처리한다. 따라서 로그, 메트릭 및 추적을 집계하기 위해 사이드카를 사용해 애플리케이션 코드 수준에서 모니터링이 이루어져야 한다.

2. What Are the Differences Between SOA and Microservices?

Service-oriented architecture(SOA)와 마이크로서비스 아키텍처 스타일은 소프트웨어 아키텍처의 진화에 있어 중요한 이정표다. 아래 다이어그램은 주요 아키텍처 스타일의 발전 과정을 보여준다.

SOA는 1990년대 후반에 엔터프라이즈 소프트웨어 시스템의 복잡성 증가를 관리하기 위해 등장했다. 2000년대 들어 SOA는 업계에서 더 많은 관심을 받고 기업에서 채택하기 시작했지만 구현 복잡성 문제에 직면했다.

그러다 2010년대에는 SOA의 한계에 대응하기 위해 마이크로서비스 아키텍처가 등장했다. 많은 대형 인터넷 기업들이 서비스를 더 작은 구성 요소로 나누기 위해 마이크로서비스를 도입하기 시작했다. 컨테이너와 오케스트레이션 도구로 인해 마이크로서비스의 개발, 배포, 모니터링이 쉬워지면서 클라우드 컴퓨팅의 발전과 함께 마이크로서비스는 더욱 탄력을 받게 된다.

차이점을 조금 더 살펴보자.

SOA 아키텍처 스타일은 일반적으로 비즈니스 기능별로 서비스를 그룹화하고 여러 애플리케이션에서 공유하는 중앙 집중식 접근 방식을 통해 세분화된 서비스를 제공한다. 마이크로서비스 스타일은 애플리케이션 컨텍스트 내에서 소규모의 독립적인 서비스가 특정 기능을 수행하는 분산된 접근 방식을 통해 세분화된 서비스를 제공한다.

커뮤니케이션 방식도 시간이 지남에 따라 진화한다. SOA는 서비스가 상호 작용할 수 있는 통일된 통신 프로토콜과 표준화된 인터페이스를 강조한다. 마이크로서비스는 다양한 통신 프로토콜과 인터페이스를 사용하며, 주로 REST 또는 메세지 큐를 기반으로 한다.

클라우드 컴퓨팅은 서비스형 인프라(IaaS)에서 서비스형 플랫폼(PaaS), 컨테이너 기반 PaaS로 발전해왔다. 따라서 마이크로서비스 기반 애플리케이션은 기본적으로 컨테이너에 배포된다.

기술 아키텍처가 변화함에 따라 조직 구조도 이를 반영한다(Conway’s law). 따라서 마이크로서비스의 경우 팀 구조에 multi-functional product한 팀이 필요하다. 각 팀은 특정 도메인에 집중한다.

3. What Problems Do Microservices Solve?

2010년대에는 몇 가지 주요 과제를 해결하기 위해 마이크로서비스 아키텍처가 등장했다.

Cloud Computing Needs New Architecture

기존의 소프트웨어 아키텍처는 클라우드 컴퓨팅이 제공하는 유연성을 충분히 활용하지 못한다. 컨테이너화 기술은 하드웨어 리소스를 더 작은 단위로 분해한다. 오케스트레이션 도구는 서비스를 쉽게 확장한다. 따라서 이러한 클라우드 컴퓨팅의 발전을 더 잘 활용하기 위해 마이크로서비스 아키텍처가 등장했다.

Decouple Services and Teams for Speed

모놀리식 및 SOA 애플리케이션은 모듈 간 종속성이 긴밀하여 인터넷 산업의 폭발적인 성장에 따라 급변하는 비즈니스 요구 사항에 적응하기 어려운 경우가 많다. 마이크로서비스 아키텍처는 각 비즈니스 도메인을 경계가 있는 컨텍스트로 분리한다. 프로덕트 팀은 특정 도메인 요구사항에 집중할 수 있다. 서비스와 팀은 사전 정의된 API 사양을 통해 소통하므로 더 빠른 기능 제공과 반복적인 시장 테스트가 가능하다.

Enable Robust Development Methodologies

모놀리식 애플리케이션의 장애는 전체 애플리케이션을 중단시킬 수 있지만, 마이크로서비스는 내결함성을 갖도록 설계되었다. 단일 마이크로서비스의 장애는 다른 서비스에서 격리된다. 또한 관리 도구는 유연한 요청 라우팅 또는 서비스 다운그레이드를 제공한다. 취약점을 사전에 테스트하기 위해 의도적으로 실패를 주입하는 카오스 엔지니어링은 이런 내결함성 설계 원칙을 극한으로 끌어올린다. 부서 간 협업을 강조하는 일련의 관행과 도구인 DevOps는 이런 카오스 엔지니어링에 자연스럽게 적합하다. 이런 방법론의 변화는 신속하고 안정적인 제품 제공을 가능하게 한다.

4. What Challenges Do Microservices Introduce?

마이크로서비스 아키텍처는 기존 아키텍처가 가진 많은 문제를 해결하고 클라우드 컴퓨팅의 발전에 부합하지만, 동시에 도전 과제도 안고 있다.

Operational Overhead

마이크로서비스 시스템에서는 컨테이너를 쉽게 프로비저닝할 수 있으며, 이로 인해 인스턴스가 폭발적으로 증가하는 경우가 많다. 적절한 도구 없이 이를 모니터링하고 디버깅하는 것은 악몽이 될 수 있다.

Increased Complexity

마이크로서비스 아키텍처는 분리된 도메인을 권장하므로 서비스 커뮤니케이션 및 관계 관리가 복잡할 수 있다. 서비스 인스턴스를 효율적으로 관리하려면 서비스 검색 및 구성 관리를 활용해야 한다. 이는 또한 통합 및 테스트에 대한 과제를 제기한다.

Organizational Transformation

콘웨이의 법칙에서 알 수 있듯이 소프트웨어 아키텍처는 조직 구조를 형성한다. 마이크로서비스 아키텍처는 그에 따라 팀을 구성해야 한다.

Distributed Transaction

각 도메인에 독립적인 데이터베이스가 있기 때문에 데이터가 더욱 분산된다. 기존 아키텍처는 데이터 일관성을 보장하기 위해 데이터베이스 트랜잭션을 활용하지만, 마이크로서비스는 동일한 목적을 위해 분산 트랜잭션이 필요할 수 있다.

Performance Overhead and Challenging Error Handling

모놀리식 애플리케이션에는 여러 모듈이 빠른 인프로세스 메소드 콜을 통해 통신한다. 장애가 발생하면 함수 호출이 즉각적인 응답을 받으므로 오류를 신속히 처리할 수 있다. 반면, 마이크로서비스는 느린 서비스 간 네트워크 통신을 사용하므로 지연 시간이 더 길어진다. 또한 마이크로서비스는 일반적으로 RPC를 통해 통신한다. 따라서 모놀리스에 비해 서비스 전반의 오류 처리를 예측하기 어렵다.

5. What Are the Basic Components of Microservices Architecture?

전형적인 마이크로서비스 아키텍처는 다음과 같은 것들을 포함한다.

Load Balancer

LB는 API 게이트웨이 앞에 위치하여 들어오는 요청을 여러 API 게이트웨이 인스턴스에 분산시킨다. 이렇게 하면 단일 게이트웨이 인스턴스가 병목 현상이 발생하는 것을 방지할 수 있다. LB가 요청을 분산하는데 사용할 수 있는 일반적인 알고리즘에는 round robin, sticky sessions, least connections 등 이 있다.

API Gateway

API 게이트웨이는 들어오는 모든 요청을 처리해 관련 마이크로서비스로 라우팅한다. 이 게이트웨이는 ID 프로바이더와 통신해 요청의 인증 및 권한 부여를 처리한다. 게이트웨이는 서비스 디스커버리를 활용해 특정 요청에 필요한 마이크로 서비스를 찾는다. 또한 요청을 백엔드로 라우팅하기 전에 요청에 필터링 규칙, 정책, 변환을 적용할 수 있다.

CDN (Content Delivery Network)

CDN은 지리적으로 분산된 서버 그룹으로, 최종 사용자와 가까운 위치에 정적 콘텐츠를 캐시해 콘텐츠를 더 빠르게 전송한다. 애플리케이션은 먼저 정적 콘텐츠를 제공하기 위해 CDN을 확인한다. 콘텐츠가 CDN에 없는 경우, CDN은 느린 백엔드 서비스에서 해당 콘텐츠를 검색하는 fallback을 처리한다.

Identity Provider

ID 프로바이더는 사용자에 대한 인증 및 권한을 처리한다.

Microservices

마이크로서비스는 특정 도메인 또는 기능 영역을 중심으로 설계 및 배포된다. 각 마이크로서비스 도메인에는 자체 데이터베이스 또는 데이터 저장소가 있다. API 게이트웨이는 REST API 또는 기타 프로토콜을 사용해 마이크로서비스와 통신한다. 동일한 도메인 내의 마이크로서비스는 내부 상호 작용을 위해 RPC를 호출해 서로 대화한다.

Service Registry and Discovery

서비스 레지스트리 및 디스커버리 컴포넌트는 마이크로 서비스의 등록 및 검색을 처리한다. 새로운 마이크로서비스 인스턴스가 스핀업되면 이 서비스에 스스로 등록된다. API 게이트웨이는 이 구성 요소를 활용해 요청을 라우팅하는 데 필요한 관련 마이크로서비스를 조회하고 찾는다.

Message Queue

메세지 큐를 사용하면 마이크로 서비스가 서로 비동기적으로 통신할 수 있다. 예를 들어 결제 서비스는 결제 거래를 완료하는 데 시간이 오래걸릴 수 있다. 이 경우 업스트림 결제 시스템은 요청 메세지를 카프카와 같은 큐에 보내고 차단하지 않고 비동기적으로 응답을 기다릴 수 있다. 이렇게 하면 장기간 실행되는 다운스트림 서비스가 완료되는 동안 업스트림 프로세스가 지연되는 것을 방지할 수 있다.

Cache

캐시는 핫 데이터를 저장하는데 사용되어 빈번한 쿼리에 대한 응답성을 향상시킬 수 있다. 하지만 마이크로서비스 아키텍처에 캐싱을 추가할 때는 캐시와 기본 데이터베이스 간의 데이터 일관성을 보장하기 위해 세심한 주의를 기울여야 한다. Stale하고 out-of-date된 캐시 데이터는 잘못된 애플리케이션 동작으로 이어질 수 있다.

Data Storage: database and object store

데이터베이스와 오브젝트 스토어는 애플리케이션의 데이터를 유지하는데 사용된다. 데이터 모델 및 확장성 요구 사항에 따라 다양한 유형의 데이터베이스 또는 데이터 저장소를 선택할 수 있다. 예를 들어, 전자상거래 애플리케이션에는 확장성 때문에 카트 항목을 저장하기 위한 Cassandra와 같은 wide-column 데이터베이스가 선택될 수 있다.

Configuration Management

설정 관리 시스템은 서비스 설정을 관리하고 배포하는 중앙 집중식 시스템이다.

Monitoring and Logging

모니터링 및 로깅을 통한 관찰 가능성은 마이크로 서비스 아키텍처를 유지 관리하는데 매우 중요하다. 마이크로서비스 시스템은 기존 애플리케이션보다 훨씬 더 많은 서비스 인스턴스로 구성되므로 문제를 식별하고 디버깅하기 위해서는 자동화된 모니터링과 분산 추적이 중요하다. 또한 지속적인 모니터링은 병목 현상을 감지하기 위한 사전 용량 계획 및 예측을 가능하게 한다.

Flow Control

흐름 제어 도구는 복잡한 마이크로서비스 시스템의 장애를 처리하는데 필요한 경우가 있다. 이런 도구는 문제 발생시 트래픽을 제한하거나 리디렉션하는데 도움이 된다. 예를 들어, Hystrix는 실패한 서비스를 격리해 마이크로서비스 간의 상호 작용을 제어한다. Sentinel은 필요에 따라 특정 마이크로서비스로의 트래픽 흐름을 제어하는 회로 차단기 역할을 한다.

Service Mesh and Service Orchestration

서비스 메시는 마이크로서비스 전반의 통신 및 제어를 처리하기 위한 인프라를 제공한다. 각 서비스 인스턴스와 함께 배포된 사이드카 프록시를 사용해 수신 및 발신 트래픽 흐름을 관리한다. 이를 통해 서비스 전반의 모니터링 및 트래픽 관리와 같은 기능을 사용할 수 있다.

서비스 오케스트레이션은 단계의 순서와 워크플로우를 조정하는데 중점은 둔다. 서비스 메시가 제공하는 기능을 활용해 마이크로서비스 상호 작용을 관리하고 워크플로우 실행을 제어한다. 구글의 Borg 클러스터 매니저의 영향을 받은 k8s는 CNCF에서 유지 관리하는 인기있는 컨테이너 오케스트레이션 시스템이다.

6. What Are Some Popular Microservice Solutions?

리눅스 재단 프로젝트인 CNCF는 컨테이너 기술을 발전시키고 클라우드 컴퓨팅을 위한 생태계를 구축하기 위해 2015년에 설립되었다. CNCF Cloud Interactive Landscape에는 도구와 성숙도가 나열되어 있다.

7. What is a Service Registry?

서비스 레지스트리는 서비스 등록 및 검색을 관리한다. 서비스 레지스트리에는 서비스 및 위치에 대한 정보를 저장하는 중앙 집중식 데이터베이스가 있다. 클라이언트 요청을 관련 마이크로서비스 인스턴스로 전송한다.

아래 다이어그램은 시작 및 종료 시 서비스가 등록되는 방식을 보여준다. 자체 등록과 써드파티 등록의 두 가지 접근 방식이 있다. 자체 등록에서는 서비스가 서비스 레지스트리에 자신의 상태를 보고하면 레지스트리 테이블이 업데이트 된다. 써드파티 등록에서는 서비스 매니저가 인스턴스를 시작 및 중지하고 레지스트리를 업데이트해 서비스의 수명 주기를 관리한다. 서비스 매니저는 또한 서비스 상태를 모니터링하기 위해 서비스를 ping 한다.

서비스 레지스트리의 또 다른 중요한 기능인 서비스 디스커버리에 대해 살펴보자. 아래 다이어그램은 클라이언트 측 검색과 서버 측 검색이라는 두 가지 서비스 검색 프로세스를 보여준다.

클라이언트 측 검색에서 클라이언트는 먼저 서비스 레지스트리에서 서비스 정보를 쿼리한다. 그런 다음 위치 조회 없이 API 게이트웨이에 요청을 보낸다. 이런 방식으로 서비스 검색 기능이 API 게이트웨이 계층 밖으로 이동한다.

서버 측 검색에서는 API 게이트웨이가 서비스 위치 찾기를 처리한다. 클라이언트 요청을 먼저 받은 후 적절한 서비스로 리디렉션한다. API 게이트웨이는 위치를 캐시해 검색 프로세스의 속도를 높일 수 있다. 그러나 이는 두 가지 측면에서 API 게이트웨이를 더욱 복잡하게 만든다.

  1. API 게이트웨이는 서비스 검색 로직을 구현해야 한다.
  2. 위치 캐시를 레지스트리 데이터와 일관되게 유지하는 것은 어려운 일이다.
