From 0 to Millions: A Guide to Scaling Your App - Part 2

Llama 5.0 캐시 추가

primary-replica 아키텍쳐를 구축한 후에는 대부분의 애플리케이션을 수십만 명의 사용자로 확장할 수 있다. 하지만 읽기 작업이 많은 일부 애플리케이션의 경우 primary-replica 아키텍쳐가 급증하는 트래픽을 제대로 처리하지 못할 수 있다. 전자 상거래를 예로 들면, 블렉 프라이데이 세일과 같은 이벤트는 데이터베이스에 쉽게 과부하를 일으킬 수 있다. 부하가 충분히 과중하면 일부 사용자는 판매 페이지를 로드하지 못할 수 도 있다.

이러한 상황을 처리하기 위한 논리적 단계는 읽기 작업을 최적화 하기 위한 캐시 계층을 추가하는 것이다. 이런 목적으로 사용되는 Redis는 자주 엑세스하는 데이터를 메모리에 캐싱하여 데이터베이스의 읽기 부하를 줄인다. 이렇게 하면 느린 데이터베이스 대신 캐시에서 데이터를 검색하므로 데이터에 더 빠르게 엑세스할 수 있다. 데이터베이스에서 수행되는 읽기 작업의 수를 줄임으로써 Redis는 데이터베이스 클러스터의 부하를 줄이고 전반적인 확장성을 개선하는데 도움이 된다.

예제 애플리케이션에서는 read-through 캐싱 전략을 사용하여 캐시를 배포한다. 이 전략은 데이터베이스에서 데이터를 읽기 전에 먼저 캐시에서 데이터를 확인한다. 캐시에서 데이터를 찾으면 즉시 반환하고, 그렇지 않으면 데이터베이스에서 로드하여 나중에 사용할 수 있도록 캐시에 저장한다.

대규모로 캐싱 계층을 배포할 때 다른 캐시 전략과 운영상 고려해야 할 사항이 있다. 예를 들어 캐시에 저장된 데이터의 다른 복사본을 통해 데이터 일관성을 유지해야 한다.

캐싱이 가능한 또 다른 종류의 애플리케이션 데이터는 이미지, 비디오, 스타일시트, 애플리케이션 번들 등 애플리케이션의 정적 콘텐츠로, 업데이트 빈도가 높지 않은 데이터이다. 이러한 데이터는 콘텐츠 전송 네트워크(CDN)에서 제공해야 한다.

CDN은 최종 사용자와 물리적으로 더 가까운 곳에 위치한 서버 네트워크에서 정적 콘텐츠를 제공하여 지연 시간을 줄이고 웹 페이지의 로딩 속도를 개선한다. 그 결과 특히 애플리케이션 서버에서 멀리 떨어져 있는 사용자에게 더 나은 사용자 경험을 제공할 수 있다.

Llama 6.0 DB 샤딩

캐시 계층은 읽기 작업이 많은 애플리케이션에 어느 정도 도움을 줄 수 있다. 그러나 계속 확장해 나가면 쓰기 요청의 양이 단일 기본 데이터베이스에 과부하가 걸리기 시작한다. 이때 기본 데이터베이스를 샤드하는 것이 합리적일 수 있다.

데이터베이스를 샤딩하는 방법에는 수평 또는 수직 샤딩의 두 가지가 있다.

수평 샤딩이 더 일반적인 방법이다. 테이블의 하나 이상의 열에 있는 값을 기준으로 여러 데이터베이스 서버에 걸쳐 데이터를 분할하는 데이터베이스 파티셔닝 기법이다. 예를 들어, 대규모 사용자 테이블을 사용자 ID를 기준으로 분할할 수 있다. 이렇게 하면 여러 개의 작은 테이블이 별도의 데이터베이스 서버에 저장되며 각 테이블은 이전에 단일 기본 데이터베이스에서 처리하던 행의 작은 하위 집합을 처리하게 된다.

수직 샤딩은 덜 일반적이다. 애플리케이션의 특정 요구 사항에 따라 테이블 또는 테이블의 일부를 다른 데이터베이스 서버로 분리한다. 이렇게 하면 각 열의 특정 액세스 패턴을 기반으로 애플리케이션을 최적화 할 수 있다.

데이터베이스 샤딩에는 몇 가지 중요한 단점이 있다.

첫째, 샤딩은 애플리케이션과 데이터베이스 계층에 복잡성을 더한다. 데이터를 여러 데이터베이스에 분할하여 분산해야 하므로 데이터 일관성과 무결성을 보장하기 어렵다.

둘째, 샤딩은 성능 오버헤드를 유발한다. 특히 여러 샤드의 데이터를 필요로 하는 작업의 경우 애플리케이션 지연 시간을 증가시킨다.

Llama 7.0 분산 Object Store

일부 애플리케이션의 경우 정적 콘텐츠가 웹사이트의 중요한 부분을 차지한다. 이러한 콘텐츠는 이미 CDN에서 제공되고 있지만 단일 파일 서버의 파일 저장 공간을 초과할 수 있다. 파일 서버를 확장하는 대신 이러한 정적 파일을 AWS S3와 같은 분산 object store에 저장하는것이 좋다. Object store는 사실상 무제한의 저장 공간과 강력한 내구성을 제공한다. 파일을 저장하고 검색하기 위한 사용하기 쉬운 API를 제공한다. CDN의 origin 서버 역할을 직접 수행할 수 있다.

Llama 8.0 모둘형 모놀리스

애플리케이션 계층과 스토리지 계층에 대한 최적화를 완료했다. 이 애플리케이션 아키텍쳐는 백만 명의 사용자를 처리할 수 있어야 한다.

사용자가 더욱 증가해 단일 서비스 인스턴스가 더 많은 부하를 처리해야 한다고 가정하면 애플리케이션은 각 서비스를 전문 팀에서 관리해야 하는 단계에 도달하게 된다. 모듈을 별도의 애플리케이션 서버에 배포된 독립적인 서비스로 분할할 수 있다. 이러한 서비스는 원격 프로시저 호출(RPC) 계층으로 래핑되며 RPC를 통해 서로 통신한다. 일반적인 RPC 프레임워크에는 gRPC, Thrift 등이 있다.

모듈형 모놀리스에서는 서로 다른 도메인 간에 엄격한 경계가 존재하며 모든 코드가 단일 애플리케이션을 구동한다. 이는 본격적인 마이크로서비스 이전의 단계이며 대부분의 사용 사례를 만족시킬 수 있다.

Llama 9.0 마이크로 서비스

서비스가 더욱 성장하게 되면 서비스 계층은 자연스럽게 더 세분화되어 진화한다. 예를 들어 서로 다른 서비스가 공통 기능을 공유할 수 있다. 소규모 팀으로 공통 기능을 관리하는 것이 이상적일 수 있다. 기능 및 서비스가 서로 다른 속도로 개발되고 서로 다른 릴리즈 일정에 따라 배포된다.

마이크로 서비스 아키텍쳐는 애플리케이션을 소규모의 독립적인 서비스 모음으로 구성하며 각 서비스는 자체 프로세스를 실행하고 gRPC와 같은 경량 프로토콜을 통해 서로 통신한다. 이 예제에서는 서비스를 웹 계층으로 분할하여 서비스가 클라이언트 사용자 요청을 처리한다.

웹 서비스는 요청을 비즈니스 계층의 적절한 서비스로 전달한다. 여기서는 사용자 서비스, 주문 서비스, 인벤토리 서비스를 예로 든다. 마이크로 서비스 아키텍쳐에서는 데이터베이스가 서비스별로 특화되는 경향이 있으며 해당 서비스를 소유한 특정 팀에서 관리한다는 점에 주목할 필요가 있다. 이 예에서는 사용자 데이터베이스 클러스터, 주문 데이터베이스 클러스터, 재고 데이터베이스 클러스터가 있을 수 있다. 요청 부하에 따라 필요한 경우 각각을 샤딩할 수도 있다.

마이크로서비스 아키텍쳐에서는 지연 시간이 중요하지 않은 요청을 메세지 큐로 전달하는 것도 일반적이다. 이렇게 하면 애플리케이션의 지연 시간이 중요한 부분과 비동기적으로 처리할 수 있는 부분을 분리할 수 있다. 전반적으로 마이크로서비스 아키텍쳐는 복잡한 애플리케이션을 더 작고 관리하기 쉬운 구성 요소로 분해하여 개발 효율성을 높일 수 있는 방법을 제공한다.
