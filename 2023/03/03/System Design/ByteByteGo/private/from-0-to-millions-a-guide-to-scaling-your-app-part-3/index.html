From 0 to Millions: A Guide to Scaling Your App - Part 3

Recent Trends

  • Cloud computing
    클라우드 컴퓨팅은 가장 기본적인 형태로 클라우드 제공업체가 관리하는 컴퓨팅 리소스에서 애플리케이션을 실행하는 것입니다. 클라우드 컴퓨팅을 사용하면 하드웨어를 직접 구매하거나 관리할 필요가 없습니다.

  • Serverless computing
    서버리스 컴퓨팅은 클라우드 컴퓨팅의 편의성에 더 많은 자동화를 더한 것입니다. 이를 통해 개발자는 클라우드 서버를 프로비저닝할 필요 없이 애플리케이션을 빌드하고 실행할 수 있습니다. 서버리스 공급자가 인프라를 처리하고 필요에 따라 컴퓨팅 리소스를 자동으로 확장 또는 축소합니다. 따라서 개발자는 확장에 대한 걱정 없이 애플리케이션 코드 자체에 집중할 수 있게 됩니다.

  • Frontend hosting platforms
    클라이언트 애플리케이션 프레임워크와 프론트엔드 애플리케이션을 쉽게 배포할 수 있는 프론트엔드 호스팅 플랫폼이 확산되고 있습니다.

Modern Frontend Frameworks and Hosting Platforms

최근 몇 년 동안 웹 애플리케이션 구축은 SPA(single page application)으로 변화해 왔습니다. SPA는 사용자가 애플리케이션과 상호 작용할 때마다 새 페이지를 로드하는 대신 현재 페이지를 동적으로 업데이트하여 보다 원활한 사용자 경험을 제공합니다. SPA에서는 초기 HTML과 해당 리소스가 한 번만 로드되고 이후 애플리케이션과 상호작용은 javascript를 통해 기존 페이지 콘텐츠를 조작하는 방식으로 처리됩니다.

기존 웹 애플리케이션들이 사용하는 MPA(multi page application)는 사용자가 링크를 클릭하거나 양식을 제출할 때마다 서버에서 새로운 HTML 페이지를 제공하는 방식입니다. 각 페이지 요청에는 일반적으로 전체 페이지 새로 고침이 포함되므로 속도가 느려지고 때로는 사용자 경험을 방해합니다.

반면 SPA는 애플리케이션의 초기 HTML 프레임을 로드한 다음 필요에 따라 서버에 데이터를 요청합니다. 이 방식은 서버가 전체 HTML 페이지를 지속적으로 전달하지 않고 정의된 API를 통해 데이터를 제공하게 함으로써 리소스를 효율적으로 사용할 수 있습니다. 또한 SPA의 API 중심 접근 방식의 또 다른 이점은 동일한 API가 모바일 애플리케이션과 공유되는 경우가 많아 백엔드 유지 관리가 더 쉬워진다는 점입니다.

SPA는 React와 같은 javascript 프레임워크를 사용하여 구축됩니다. 이런 도구들은 성능과 유지보수에 최적화된 복잡한 애플리케이션을 구축하기 위한 일련의 추상화 및 도구를 제공합니다. 반면 MPA를 구축하기 위해서는 서버 중심적인 접근 방식이 필요하며 애플리케이션의 복잡성이 증가함에 따라 확장 및 유지 관리가 더 어려워질 수 있습니다.

이런 클라이언트 프레임워크의 인기가 높아지면서 다양한 프론트엔드 호스팅 플랫폼이 개발되었습니다. 이런 플랫폼은 개발자가 코드를 리포지토리에 체크인하면 호스팅 플랫폼이 웹 애플리케이션 번들과 관련 리소스를 자동으로 빌드하여 CDN에 배포합니다. 이러한 호스팅 플랫폼은 클라우드 및 서버리스 컴퓨팅 기반에 구축되어 사용자와 가까운 엣지에서 데이터를 제공하는 등의 모범 사례를 보여줍니다.

Modern Backend Options

이전에 언급했듯이 최신 벡엔드의 역할은 프론트엔드 웹 및 모바일 애플리케이션을 지원하기 위해 잘 정의된 일련의 API를 제공하는 것입니다. 리소스의 제약이 있는 소규모 스타트업이 백엔드를 구축할 때 무엇을 사용해야 하는지, 그런 백엔드를 얼마나 확장할 수 있는지 살펴봅니다.

애플리케이션이 마켓에 출시되는 시간이 중요하고 리소스가 제한되어 있는 스타트업은 비핵심 업무를 최대한 많이 덜어내야 합니다. 서버리스 컴퓨팅 옵션은 아래와 같은 이유로 긍정적으로 검토될 수 있습니다.

  • 서버리스 컴퓨팅은 확장, 이중화, 장애 조치와 같은 백엔드의 운영 측면을 관리하므로 스타트업 팀은 인프라 관리에서 벗어날 수 있습니다.
  • 서버리스 컴퓨팅은 비용 효율적인 사용량 기반 요금 모델을 따릅니다.
  • 서버리스 컴퓨팅을 사용하면 개발자는 서버 관리에 대한 걱정 없이 코드 작성과 백엔드 테스트에만 집중할 수 있습니다.

Serverless Computing Options

스타트업에 적합한 서버리스 컴퓨팅 옵션에는 cloud funtions와 managed container platforms가 널리 사용됩니다.

Cloud funtions는 간단한 애플리케이션 로직을 처리하도록 설계된 서버리스 컴퓨팅 플랫폼입니다. 요청량에 따라 자동으로 확장되고 간단한 요청/응답 모델을 따릅니다. 그러나 Cloud funtions는 일반적으로 소수의 인기있는 프로그래밍 언어로 제한됩니다.

Managed container platforms는 cloud funtions와 유사합니다. 가장 큰 차이는 더 많은 런타임 환경을 지원하기 때문에 대부분의 프로그래밍 환경을 자유롭게 선택할 수 있습니다. 관리형 컨테이너 플랫폼의 프로그래밍 모델은 cloud funtions보다 유연하고 장기적인 stateless server로 운영되며 컨테이너 간에 부하 분산과 확장, 실패 복구를 자동으로 처리합니다.

API Gateway

비용을 더욱 절감하기 위해선 서버리스 컴퓨팅 옵션 앞에 API 게이트웨이를 배치하는것을 고려하는게 좋습니다. API 게이트 웨이는 인증 및 권한 부여를 처리할 수 있도록 구성이 가능합니다. 또한 모니터링 속도 제한 및 캐싱과 같은 기능을 제공하여 성능과 비용 절감에 도움이 될 수 있습니다.

Serverless Database Options

다양한 종류의 선택 가능한 데이터베이스 옵션이 마켓에는 있습니다. 적절한 데이터베이스를 선택하기는 어려운 일 중에 하나이지만, 몇가지 참고할 수 있는 지침이 있습니다.

스타트업에서는 여러가지 이유로 관계형 데이터베이스를 사용해야 한다고 생각합니다.

첫째, 관계형 데이터베이스는 잘 정립된 기술입니다. 관계형 데이터베이스는 스키마 유연성이 뛰어나 빠르게 실험할 수 있습니다. 그리고 NoSQL에 비해 방향을 변경하는데 마찰이 적습니다. NoSQL을 사용하면 데이터 스키마가 더 엄격해집니다. NoSQL의 경우 모든 데이터 엑세스 패턴을 미리 결정해야 합니다.

둘째, 최신 관계형 데이터베이스는 대규모로 실행 가능합니다. 대부분의 API 작업 부하에서 관계형 데이터베이스가 수십만 명의 활성 사용자를 처리하는 경우가 드물지 않습니다.

셋째, 관계형 데이터베이스의 한계에 도달한 경우 cache나 replication을 도입하는 등 데이터베이스 계층을 확장하는 well-known 방식들이 있습니다.

서버리스 관계형 데이터베이스는 리소스 비용과 관리적인 측면에서 긍정적인 면이 있습니다. 하지만 반대로 비교적 새로운 개념이기 때문에 클라우드 업체간에 제품 성숙도가 다릅니다. 특정 업체에 좋은 제품이 없다면 관리형 데이터베이스도 좋은 선택지가 될 수 있습니다. 클라우드 제공 업체가 지속적으로 데이터베이스의 소프트웨어, 서버 하드웨어의 유지 관리를 처리하고, 사용자는 특정 작업 부하에 대한 확장, 백업 및 데이터베이스 튜닝과 같은 일부 운영 작업만을 담당합니다.

소프트웨어 엔지니어링의 많은 부분과 마찬가지로 데이터베이스의 선택은 장단점을 고려해야 합니다. NoSQL이 더 나은 선택지가 될 수 있지만 제품 / 시장 적합성이 불확실하고 최대한의 유연성을 유지하는 것이 중요한 대부분의 스타트업의 경우 관계형 데이터베이스가 더 나은 선택일 가능성이 높습니다.

Modern Startup Stack

프론트엔드 호스팅 플랫폼, 서버리스 API 백엔드, 서버리스 관계형 데이터베이스를 통해 많은 노력을 미리 투자하지 않고도 많은 활성 사용자를 처리할 수 있는 스택으로 시작할 수 있습니다. 이 스택은 인프라나 서버를 관리하지 않고 서비스 내부에 고가용성 및 다중 영역 이중화 기능이 포함되어 있습니다. 트래픽이 증가함에 따라 서비스를 확장할 책임은 서비스 제공 업체에 있습니다. 수만 명에서 일부는 수십만 명의 사용자를 처리할 수 있으며 단일 서버로 달성할 수 있었던 것과 비교할 수 없습니다.
